## 查询执行引擎总体设计

### Overall Process

类似于[`bustub`](https://github.com/cmu-db/bustub), 查询执行引擎分为五部分：

`Parser` -> `Preprocessor` -> `Planner` -> `Optimizer` -> `Executor`

`Parser`: 解析`sql`语句得到`AST`

`Preprocessor`: 绑定`AST`中各实体，并通过此种方式进一步检验`sql`语句的正确性

`Planner`:  生成初始逻辑查询计划

`Optimizer`: 实现基于规则的优化，实现`selection`的下移

`Executor`: 生成物理查询计划，并执行得到结果

### Parser

该项目使用c++为佳，因此选用flex+bison完成语法解析。

基本要求：

1. 文法支持数据类型：字符串、整形、时间类型
2. 支持基本表的查询
3. 支持两个表的等值连接

### Preprocessor

`Preprocessor` 部分有三个任务：

1. 绑定实体

   将关系和属性（当前存储为字符串）绑定至具体的关系与属性，但这部分超出了查询引擎的范围，因此这部分需要设计接口待实现，要求尽量**通用**与**精简**。

2. 语法检查

   1. 关系是否存在
   2. 属性是否存在，且无二义性
   3. 数据类型是否与属性一致

3. 简化条件子句：

   1. 拆分成`and`相连的条件分句

### Planner

将绑定好的`AST`简单转换为逻辑查询计划树即可，支持逻辑算子：

* projection
* filter
* table_scan
* join

### Optimizer

实现基于规则的优化：

1. 执行引擎内默认开启选择下推优化
2. 提供一个register方法，使用户能够添加自己的优化方案
3. 若有时间尝试添加子查询优化（教材上），但当前子查询语法还未设计

### Executor

将优化好的逻辑查询算子转换成物理算子，物理算子以迭代器模型为基类，通过`getNext()` 获取所有数据。

对应逻辑算子的物理算子：

* projection
* filter
* table_scan
* join

这其中table_scan需要与存储部分交互，以获取缓冲区的内容。
